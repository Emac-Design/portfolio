Re-designing Maidstone Borough Council's Website

Background

The website was initially re-designed and re-built from scratch and published back in February 2018.

Whilst this design was a drastic improvement to the preious design - it wasn't designed and built on user needs.

Some user needs were used to influence the design and build but the main drive was to give a new lease of life by using GOV.UK as it's foundation.

The sites accessibility was greatly improved but it still fell short of where it needed to be. The new WCAG 2.1 guidelines came in Sepetmber 2019.

The decision was made to re-design the website once more to fix the accessibility issues to comply to WCAG 2.1 and to make sure it meets user needs.

The first iteration was designed and developed by a previous member of them team. You can view that design here. Some of the user needs haven't been met as we didn't conduct any user research with Councillors or staff.

This lead to multiple iterations of this design.

User research and user needs

There were several types of users defined in this project. They included the residents of the borough, residents outside of the borough as well as internal users such as staff and Councillors

Some of the user needs are listed below.

External user needs

As a user using a mobile, I want a link back to the top of the page, so that I don't have to scroll back to the top.

As a user, I want to find common tasks, so that I don't have search through the site to find them.

As a user, I want won't use an A to Z, as the terms I use won't match the terms the Council uses.

Internal user needs

As a Councillor, I want the site to have a clear brand with our logo, so users are clear what site their on.

As a Councillor, I need our website to be visually appealing, so that I'm happy to refer to it.

As a member of staff, I need the website to be easy to navigate, so that I can tell someone on the phone how to find something.

As a member the Digital Services team, I need the website to be accessible, so that we're compliant with WCAG 2.1.

After conducting user research with our Counicllors, they thought that the search bar on the first design was too prominent as they didn't use it or found that it didn't work the way they expected it to.

They also thought the design wasn't visually appealing. They commented that other local authority sites were more appealing making them want to engage more with the site. The thing these sites had in common was icons.

Design and interations

Further adaptations of the design were created through Adobe XD using a 12 column grid as the final design will be developed using Bootstrap.

Top tasks

User research showed that they was demand for a top tasks section. This is an area where a user can easily find transactions they're most likely to use without having to navigation through to sections of the site.

The research made it clear that this section needed to sit above our main services area.

Visual appearance

Search bar

User research showed that there were issues with the search. I drastically decreased its size and placed it in the top right hand corner of the navigation bar.

Service area listing

This area has had serveral iterations. Feedback from the coded design said that users weren't clear that each service area had more than three subsections within it.

Several options we're explored to overcome this issue. One included removing all options whilst purely having the heading as a link with an icon to the left of it. The most recent iteration will combine the two as users felt in the dark as to what the service area heading offered.

A to Z

One of the user needs for the re-design made it clear that the A to Z wasn't used.

It takes up a lot of space on our current design so I've shrunk this down completely by removing the alphabet listing and creating small section above the footer that contains a to the A to Z page.

Micro interactions

Links

Our current design doesn't always underline links - especially on our alerts so a user doesn't always know that they can interact with that element.

This has fixed by underlining all links across the site.

Hover states

On the current design the hover state on link change from our corporate blue to a pink. The shade of pink fails colour contrast ratios for WCAG 2.1 so this needed to be updated.

The hover state on the new design will change to a darker shade of our corporate blue.

Focus states

On the current design there are no focus states on an interactable element.

The new design does have focus interactable elements. They're based off GOV.UK approach using a yellow background with black text.

CSS

This new design is using SASS file rather than a conventional CSS file.

I wanted to use SASS for a few reasons. Firstly, to keep up with modern development trends but also to improve the sites web performance. Another reason is to improve the skills within our team - everyone will learn a new skill.
