Report Litter

Background

The contract for our reporting forms (like Report Litter) was due to expire at the end of October 2019. These used a different forms platform called Abavus.

This was one of a eight forms that needed to be rebuilt into our main forms platform (Achieve Forms). Of these eight forms, I built 3 of them.

The old forms platform had workflow built into them so that our Street Cleansing team can view all the information given by a resident.

User needs

Some of the user needs are included below.

External user needs

As a resident, I want to show you photos of the issue, so that you know exactly where the problem is.

As a resident, I want to be updated on my report by text, so that I don't have to chase you for the outcome.

As a resident, I want show you where the litter is on a map, so that you know exactly where it is.

Internal user needs

As part of the Street Cleansing Team, I need to sign a report off, using a mobile device.

As part of the Street Cleansing Team, I need a map showing me where the report is, so that I can find it easily.

As the manager of the Street Cleansing Team, I need to see whether requests are being signed off in time, to make sure we meet our Service Level Agreement (SLA).

Design and interactions

This has been in live beta since 01/10/2019 and has one of our highest form completion rates at 75%. This evidences that this new process is a drastic improvement from the previous form.

The map

The first version of the form included a map to locate the area where the litter or litter bin is. However, there was no manual search for the map, meaning that a user would have to locate the area purely using their fingers. This was a major accessibility issue but it also hindered the usability of the map as we have a lot of public litter bins in rural locations - local knowledge would be needed to find them.

Component based design

Inspired by GDS, this was the first project where I designed and built re-usable components.

Three re-usable components were created based off user needs. A single pin map, an "About you" page and an "Updates" page. These three pages feature in all eight reporting forms. Designing this way sped up production time as well as ensuring a consistent design across all eight forms.

Providing updates

The next iteration will involve more detailed text messages as to why we have dealt with or cannot deal with their report.

Workflow

A major part of my work at the beginning of this project was to map out the first stage of the process that a user would complete.

I started with a list of questions that we're needed by the Street Cleansing team as part of the their internal user needs. Each page of the forms became the post it notes - making it easy to re-arrange the flow.

Part of the building this user centered process involved building a back end work flow for our Street Cleansing Team to sign off each report once it was completed.

The screenshot below is the work flow that a report follows. It starts with the initial report which gets sent to our Street Cleansing crews dashboard. Once they've completed the report the customer will be notified of the outcome. At this stage they'll have to opportunity to escalate the report to a supervisor if they're unhappy with the outcome. The supervisor can then send the job back to Street Cleaning crew or close it down as a completed case.

Design system

I'll also be looking at the design of all the reporting forms to improve the way the we use alerts and in-page notifications. There's currently no best practise or guidelines on when and how to use them - this will be the first component of our corporate design system that I'm leading on.

The future

I'll be looking into the way that cases are assigned to the crew. It's a manual job that the superviser has to do so the next iteration I'll look at ways of distrubing reports based on the location so remove a complication from the process.

One of the re-occuring user needs wanted previous reports plotted on a map so that they know whether to report it or not.

I'm in the process of building a database lookup which will run when the map has loading. This look up will find all open cases within the last fourteen days and then populate those open cases on the map. When the marker is opened - it'll tell you what type of report it is and when it was reported.

The lookup is working and populating into the map, however, futher research is needed as an existing case plotted on the map has a different coloured marker to the user selecting a location in the map.
